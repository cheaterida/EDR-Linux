# v0.7 Rootkit 检测插队决策记录

> 日期: 2026-06-17 | 状态: 已批准，Phase 0 文档同步完成 | 基于: DESIGN.md + v0.6-decision-record.md + v0.7-plan.md

## 背景与假设变更

### 原假设（v0.6 决策）

v0.6 决策记录中明确跳过以下探针：

| 探针 | 原始优先级 | 跳过理由 |
|------|-----------|---------|
| `bpf()` syscall | P0 | 学生红队极少操纵 BPF |
| `init_module` / `delete_module` | P0 | 学生红队极少加载内核 rootkit |
| `execveat` | P1 | 学生红队几乎不用 execveat 绕过 exec 探针 |
| `mount` / `move_mount` | P1 | 演习未用容器 |
| `nsenter` | P1 | 演习未用容器 |

当时判断：学生红队主要使用 curl/bash/python/nc 等常见工具，不会走内核 rootkit 路径。防线依赖 systemd `ProtectKernelModules=true`。

### 新信息（2026-06-17）

指导教师在演习前明确：**红队极有可能采取 rootkit 从内核层攻击**。原假设不再成立，必须在 v0.7 窗口内补上 rootkit 检测与响应能力。

## 决策结果

### 覆盖维度（五维狩猎面）

| 维度 | 优先级 | 是否纳入 1 周 | 实现方式 |
|------|--------|--------------|---------|
| 模块加载/卸载 | P0 | ✅ | eBPF tracepoint `init_module`/`finit_module`/`delete_module` |
| eBPF 操作监控 | P0 | ✅ | eBPF tracepoint `bpf()`，重点监控 BPF_PROG_DETACH / BPF_LINK_DETACH |
| 进程隐藏（DKOM） | P0 | ✅ | /proc 遍历 vs BPF 观测 PID 集合跨源校验 |
| 模块隐藏 | P1 | ✅ | /sys/module/ 遍历 vs /proc/modules 跨源校验 |
| 网络隐藏 | P1 | ⚠️ 可选 | ConnTracker vs /proc/net/tcp 对比，时间允许再做 |
| 文件隐藏 | P2 | ❌ 延后 | getdents hook 跨视图对比 |
| syscall table hook | P2/P3 | ❌ 延后 | 内核内存扫描过于脆弱 |

### 响应策略

- 默认 **monitor 模式**，方便测试；演习前可切换 enforce。
- enforce 时优先 `network_isolate`（nftables 层，超出 rootkit 常见 kill hook 范围），必要时 `kill`。

### 与 v0.7 原规划的协调

v0.7 原规划重心是运维审计可用性（report generate、事件归并、web 清理）。现在双轨并行：

- **轨道 A（rootkit）**：前 3 天主攻 P0 探针 + 跨源校验。
- **轨道 B（运维审计）**：后 2 天做 report generate 和过滤字段补齐；web 面板删除压到最后半天。

如果 1 周时间紧张，优先保证 rootkit P0 落地，v0.7 的 report generate 可简化。

## 技术方案要点

1. **新增 `internal/rootkit/` 包**：跨源校验核心，定时扫描。
2. **新增 2 个 BPF C 源文件**：
   - `internal/bpf/probes/module.bpf.c`
   - `internal/bpf/probes/bpfop.bpf.c`
3. **新增事件类型**：`EventModuleLoad`、`EventModuleUnload`、`EventBPFOp`。
4. **新增 category**：`"rootkit"`，规则引擎 `Validate` 允许。
5. **复用现有管线**：`Logger.Write` + `Responder.Apply`，不改动事件模型。
6. **编译方式**：沿用现有 Makefile 现场编译（与原有探针一致）。
